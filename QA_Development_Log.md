# GridWorld 專案開發對話與關鍵技術紀錄 (Q&A Log)

> 這份紀錄還原了我們在建構「強化學習 GridWorld 視覺化平台」過程中的關鍵討論點，包含了架構決策、數學演算法實作細節，以及前端效能最佳化策略。

---

**User (學生):** 
在沒有使用 OpenAI Gym 的情況下，我們該如何在純 Python 中有效率地實作馬可夫決策過程 (MDP) 的狀態轉移與邊界碰撞條件？這個部分的邏輯如果沒寫好，會導致 Agent 跑出格子外。

**Assistant (AI):**
這是一個非常關鍵的起點。為了滿足完全從零開始的限制，我們在 `rl_environment.py` 中建立了一個自定義的 `GridWorldEnv` 類別。
關於狀態轉移，我們定義了四個基底向量：`directions = [(-1, 0), (1, 0), (0, 1), (0, -1)]`（上、下、右、左）。
當給定當前座標 `(r, c)` 與 `action` 時，我們會計算出擬態的新座標 `(nr, nc) = (r + dr, c + dc)`。
接著透過一個內建的 `_is_valid(r, c)` 去檢查：如果 `0 <= nr < n` 且 `0 <= nc < n` 且該格子**不在障礙物串列中**，這步才是合法的。如果發生越界或撞到障礙物，狀態就會維持在原點 `(r, c)`，同時回傳撞擊懲罰 `-1.0`。這樣的判定方式在 $O(1)$ 的時間複雜度內就能完成，對於疊代百萬次的動態規劃來說非常高效。

**User (學生):** 
了解。那麼在實作價值迭代 (Value Iteration) 時，也就是執行 Bellman Optimality Equation，我們如何確保演算法能正確收斂？而且最後提取出最佳策略 (Greedy Policy) 的細節是什麼？

**Assistant (AI):**
價值迭代的核心在於「利用下一狀態的最大可能價值」來更新當前狀態的價值。
在實作上，我們設定了一個極小的收斂閾值 `theta = 1e-6`。每次進入迴圈時，我們建立一個變數 `delta = 0` 來追蹤整個矩陣中「單一格子價值變動的最大絕對值」。
我們遍歷整個網格（除了終點與障礙物），並計算上下左右四個動作的期望回報：`v = reward + gamma * V[nr, nc]`。找到這四個動作中最大的 `v` 賦予格子。
如果這個回圈結束時 `delta < theta`，代表價值收斂，演算法自動跳出。
而在收斂後，我們會再遍歷一次尋找每個格子的 `argmax(a)`，並將最大的動作標記為對應的 Unicode 箭頭符號（↑、↓、→、←），這就是最終提取出來的貪婪動作（Greedy Policy）。

**User (學生):** 
我希望這個介面在滑鼠操作上能非常直覺，像是「拖曳滑鼠連續繪圖」。但我知道如果在 React 等框架下，不斷更新 81 個格子的狀態可能會產生 Virtual DOM 的效能損耗。在 Vanilla JS 實作上有什麼關鍵能避開這個問題？

**Assistant (AI):**
非常好的效能考量。為了解決這點，我們在 `main.js` 中採用了 **「事件委派 (Event Delegation)」** 模式。
我們**沒有**在 81 個 grid-cell 上分別綁定監聽器，而是在它們的父級容器 `grid-container` 上單一綁定了 `mousedown`、`mouseover` 與 `mouseup` 三個事件。
當使用者按下滑鼠 `mousedown`，我們設定一個全域變數 `isDrawing = true` 並觸發第一筆繪圖。隨著滑鼠移動觸發 `mouseover`，如果此時 `isDrawing` 為 true 且當前工具是「障礙物」，程式就直接竄改該格子的 DOM class (`cell.classList.add('obstacle-node')`)。
透過直接操作實體 DOM 並結合 CSS 的硬體加速渲染，完全繞過了框架常見的 diffing overhead（差異比對損耗），確保了繪圖的絲滑流暢度。

**User (學生):** 
最後一個重點，關於視覺化熱力圖， DP 值包含了包含懲罰的負數（例如靠近邊緣或障礙物），以及靠近終點的正數。我們該如何數學化地將這些數值轉換為指定的「海洋藍到亮白色」的 HSL 色碼？而且要如何兼顧字體的對比度？

**Assistant (AI):**
要將任意區間的連續數值映射到顏色上，我們必須先做**正規化處理 (Normalization)**。
在 `renderHeatmap` 函式中，程式先掃描整個矩陣找出當下價值的最小值 `vMin` 與最大值 `vMax`。
接著利用公式 `(val - vMin) / (vMax - vMin)`，將所有的數值嚴格壓縮到 `0.0 ~ 1.0` 的區間。
您指定的 HSL（色相、飽和度、亮度）模型中，海洋藍的色相為固定值 `Hue = 210`，飽和度設為 `80%`。
我們將剛才得到的 `0~1` 數值，線性映射到「亮度 (Lightness)」上，範圍設定在 `15%`(深藍黑) 到 `90%`(冰冷亮白)。
此外，我們加入了一行重要的自動對比度邏輯：`cell.style.color = lightness > 60 ? '#000000' : '#ffffff';`
當格子太亮時，文字會自動變成黑色，而在深色區域則保持白色字體，確保所有的 DP 數值浮點數都能清晰可見。

**Assistant (AI):**
沒錯！這是新手在接軌正式環境常忽略的一環。
如果您只用 `python app.py` 啟動，Flask 內建的伺服器其實只設計在開發時使用（Development Server），它只能單執行緒處理少部分請求。
這也是為何我在剛剛協助您更新了 `requirements.txt` 並加入了 `gunicorn==21.2.0`。
Gunicorn (Green Unicorn) 是一個 Python WSGI HTTP Server，它是專為並行能力高、效能強的量產環境設計的。因此在 Render 或是其他雲端部署時，我們使用指令 `gunicorn app:app` 取代原本的啟動指令，這樣就能確保就算有多個教授或助教同時造訪您的專案，也能夠即時且穩定的回傳動態規劃的運算結果。
